--- ## ⚡ Ключевые архитектурные решения ### Что мы решили и почему
**1. Layout-based routing вместо page-level auth** ```typescript // ✅ Правильно: AuthGateLayout, ProtectedLayout, RootLayout // ❌ Неправильно: Проверка auth на каждой странице ``` **Почему:** Централизованная логика, DRY, легче поддерживать
**2. Zustand вместо Context API для auth state** **Почему:** Меньше boilerplate чем Redux, персистентность из коробки, проще чем Context для глобального состояния
**3. Markdown для контента уроков** **Почему:** Разделение контента и кода, легко обновлять через Firebase, подсветка синтаксиса
**4. Design system с семантическими токенами** ```css /* ✅ Семантические токены */ --color-primary: #fbbf24; --color-bg-dark: #1a1a1a; /* ❌ Хардкод цветов */ background: #fbbf24; ```

--- ## 🔥 Конфликт #1: Routing Collision
**Проблема:** Anna использовала `/topics/:id` для learning flow, но этот маршрут уже использовался в Dashboard. Всё сломалось.
### Как обнаружили: - Anna начала implementing learning flow - Обнаружила что reuse `/topics/:id` ломает Dashboard - Topics и lessons были смешаны в Firebase ### Решение (Митинг #6):
``` ❌ Не переиспользовать /topics/:id ✅ Создать отдельный /learn/:topicId ✅ Разделить topics (метаданные) и lessons (контент) ```
### Урок: > **Не патчить проблемы — решать структурно.** > Отдельные маршруты для разных сущностей. --- ## 🔥 Конфликт #2: Временные заглушки vs Архитектура
**Проблема:** Sergey строил Firebase + Zustand архитектуру, а тиммейты создавали временные stubs, которые конфликтовали с его структурой.
### Примеры проблем: - **Два файла `App.tsx`** в разных папках - Routing Анны не работал из-за дубликата - Временные компоненты мешали интеграции ### Решение:
- Постепенная замена stubs → централизованная архитектура - Code review перед мержем - Синхронизация на митингах
### Урок: > **"Временное" = "технический долг".** > Лучше сразу делать правильно, даже если медленнее. --- ## 🔥 Конфликт #3: Дизайн-хаос
**Проблема:** Без единого design system каждый делал свой стиль: разные цвета, отступы, тени.
### Как выглядело: - Каждый компонент в своём стиле - Нет консистентности между страницами - Хардкод цветов везде ### Решение (Митинг #4):
``` 1. Просмотрели ~10 вариантов дизайна 2. Выбрали: dark theme + grey/yellow accents 3. Artem стал owner design system 4. Tailwind config с семантическими токенами ```
### Урок: > **Design system нужен с Day 1, не после.** > Один человек отвечает за консистентность. --- ## 📡 Процессы: Как мы работали ### Инструменты | Инструмент | Использование | |------------|---------------| | **GitHub Issues + PRs** | Основной workflow, code review | | **Google Meet** | Синхронизация 1-2 раза в неделю | | **Telegram** | Быстрые вопросы, не для сложных решений | | **Jira** | Первоначально, потом перешли на GitHub | ### Митинги: 8 за 6 недель - **Частота:** Раз в 3-5 дней - **Agenda:** Прогресс → Блокеры → Решения - **Формат:** Обсуждение архитектуры, review PRs

--- # ❓ Q&A ### Вопросы по архитектуре, коду и решениям --- ## 📋 Ожидаемые вопросы ### По архитектуре **1. Почему Firebase, а не PostgreSQL/MongoDB?** - Быстрый старт для прототипа - Built-in Auth + Firestore из коробки - Не нужен отдельный backend - Real-time обновления **2. Почему Zustand, а не Redux/Context API?** - Меньше boilerplate чем Redux - Персистентность из коробки - Проще чем Context для глобального состояния - TypeScript-friendly API **3. Как работает auth guard на routing уровне?** ```typescript // Layout-based routing AuthGateLayout → проверяет auth status → authenticated? → ProtectedLayout → guest? → GuestOnlyLayout → loading? → spinner ``` **4. Почему не HashRouter для GitHub Pages?** - Визуально непривлекательный (`/#/dashboard`) - Нашли решение: копировать `index.html` → `404.html` - Сохранили красивый URL --- ## 📋 Ожидаемые вопросы ### По коду **1. Покажите самый сложный компонент** `QuizPage.tsx` — 6 состояний, навигация, score calculation, results view **2. Как тестируете Firebase integration?** Mock Firebase functions (`vi.mock()`), тестирование сервисного слоя отдельно **3. Как обрабатываете ошибки API?** - Try/catch в сервисных функциях - Error state в компонентах - Toast notifications для пользователя **4. TypeScript strict mode — были ли проблемы?** - Да, особенно с Firebase types - Type assertions (`as Omit`) - Строгий режим с начала = меньше багов потом --- ## 📋 Ожидаемые вопросы ### По процессам **1. Как разрешали merge conflicts?** - Коммуникация: обсуждали конфликт до мержа - Code review: оба смотрели изменения - Git: `git pull --rebase`, ручное разрешение **2. Кто принимал финаальные решения?** - Архитектура: совместно (Sergey + Anna) - Дизайн: Artem (как owner design system) - Backend: Sergey (Firebase expertise) - Frontend: Anna (routing, learning flow) **3. Что бы сделали иначе?** - Больше синхронизаций втроём - Data structure с начала - Design system с Day 1 - Больше тестов **4. Как оценивали задачи?** - Интуитивно, по сложности - Без story points (не использовали Jira активно) - Ориентировались на дедлайны checkpoint'ов --- # 🐱 Спасибо! ### Learning can be serious and cute at the same time. --- **Деплой:** https://dzzuze.github.io/SAA-Cat/ **GitHub:** https://github.com/dzzuze/SAA-Cat **Видео Week 7:** https://youtu.be/uDiuYctqrG4
